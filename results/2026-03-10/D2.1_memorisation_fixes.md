## Actual Suspects (Re-Ranked)

### 1. OFF-BY-ONE BUG IN SAMPLE SHAPE (HIGH — newly discovered)

**Training** builds x as:
```python
cat_states_from = (B, frame_stack, transition_dim)      # from dataset
cat_states_to   = (B, chunk_size, transition_dim)        # from dataset (states_to[:,:-1,:])
x = cat([cat_states_from, cat_states_to], dim=1)         # → (B, frame_stack + chunk_size, transition_dim)
```

**Sampling** uses:
```python
shape = (num_samples, self.cfg.chunk_horizon + self.cfg.frame_stack - 1, self.transition_dim)
```

If `chunk_horizon == chunk_size` (which it should be), then:
- **Training sequence length:** `frame_stack + chunk_size`
- **Sampling sequence length:** `chunk_horizon + frame_stack - 1 = frame_stack + chunk_size - 1`

**Sampling is 1 timestep shorter than training!**

| Config | Training length | Sampling length | Mismatch |
|--------|----------------|-----------------|----------|
| D2 (fs=2, cs=8) | 10 | 9 | -1 |
| 500k (fs=1, cs=7) | 8 | 7 | -1 |

The UNet's convolutions can handle any sequence length, so this doesn't crash — but the model was trained on length-N sequences and is asked to generate length-(N-1) sequences. This is a different distribution.

**To verify:** Check the SOPE reference — does it use the same length for training and sampling?

**Fix:** Change sample shape to `(num_samples, self.cfg.total_chunk_horizon, self.transition_dim)`.

### 2. MODEL CAPACITY (MEDIUM-HIGH)

D2 used dim_mults=(1,2) → ~250k params. SOPE's Hopper uses (1,2,4,8) → much larger. Even though a 250k-param model "should" memorise 17 chunks (3,060 values), the TemporalUnet architecture may need more capacity to learn the denoising function across 256 noise levels.

The SOPE training script uses `dim_mults=(1,)` which is even SMALLER — but that script uses `action_weight=1` (vs our 5.0) and may have other differences. The actual evaluated models use (1,2,4,8) or (1,4,8).

### 3. predict_epsilon (MEDIUM — downgraded from HIGH)

Still worth testing since HalfCheetah uses False and the Diffuser class defaults to False. x0-prediction gives the model a more direct learning signal (predict the clean data, not the noise). But SOPE's Hopper works with epsilon prediction, so this alone shouldn't cause total failure.

### 4. EMA (LOW-MEDIUM)

SOPE uses EMA during training (`ema = EMA(diffusion_model)`). We don't. EMA provides a smoothed version of model weights for inference, which typically improves sample quality. For a memorisation test this probably doesn't matter, but it's easy to add.

### 5. CONDITIONING VERIFICATION (LOW — sanity check)

Verify that `make_cond()` produces the right conditioning dict and that `apply_conditioning()` in the GaussianDiffusion code pins the correct timestep indices.

### Key Finding: predict_epsilon is THE critical variable

**predict_epsilon=True completely fails** — state L2=844, essentially identical to D2's 911 despite 15x more parameters. The larger model made no difference.

**predict_epsilon=False achieves near-perfect memorisation** — state L2=0.019 (48,000x improvement over D2), loss converges to 0.017 (vs 0.557 for eps=True at the same epoch).

This is a **night-and-day difference** on the same architecture, same data, same training. The only change is whether the model predicts noise (eps) or clean data (x0).

### Why predict_epsilon=True fails

With epsilon prediction, the model predicts the noise ε that was added at timestep t. The actual clean sample is recovered via: `x0 = (x_t - √(1-α̅_t) · ε) / √(α̅_t)`. This indirect path means:
- Small errors in predicted ε get amplified by `1/√(α̅_t)` at high noise levels (large t)
- The model must learn a precise mapping from noisy input to noise — but at high t, the input is nearly pure noise, making this very hard
- With only 17 chunks, the model can't learn this mapping well enough

With x0 prediction, the model directly outputs its best guess of the clean data. At high noise levels it may be inaccurate, but the prediction stays bounded and meaningful. The loss directly penalises distance from the clean target, giving clearer gradients.

### Stitching Quality (eps=False)

The stitched trajectory stays perfectly in distribution:
- **Blowup ratio: 1.0x** (synthetic range matches real range exactly)
- All dimensions track the real trajectory closely
- Marginal statistics (mean, std, min, max) match per-dimension
- The 100% OPE relative error is actually correct: this particular trajectory didn't succeed (real return = 0), and the synthetic trajectory correctly reproduces this (synthetic return = 0). The oracle V^π = 0.54 is computed over 50 rollouts, not this one.

### Source Code Fix

Fixed off-by-one in `SopeDiffuser.sample()` (`sope_diffuser.py:208`):
```python
# Before (bug):
shape = (num_samples, self.cfg.chunk_horizon + self.cfg.frame_stack - 1, self.transition_dim)
# After (fix):
shape = (num_samples, self.cfg.total_chunk_horizon, self.transition_dim)
```
This didn't affect D2's results (chunk eval bypasses `sample()`), but would have caused incorrect sequence lengths for any code calling `sample()` directly.

## Conclusion

**The memorisation test PASSES with predict_epsilon=False.** The model architecture (TemporalUnet with dim_mults=(1,4,8)) is sound. The root cause of all previous failures was `predict_epsilon=True`.

### What changed vs D2

| Change | Impact |
|--------|--------|
| predict_epsilon: True → **False** | **48,000x improvement** in chunk L2 (THE fix) |
| dim_mults: (1,2) → (1,4,8) | No impact with eps=True (844 vs 911). Needed for eps=False capacity. |
| frame_stack: 2 → 1 | Minor (enables dim_mults=(1,4,8) via total_horizon=8) |
| chunk_size: 8 → 7 | Minor (enables dim_mults=(1,4,8) via total_horizon=8) |
| sample() off-by-one fix | No impact on this test (only affects direct sample() calls) |
