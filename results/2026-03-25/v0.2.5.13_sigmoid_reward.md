# v0.2.5.13: Sigmoid Reward Model for Noise-Robust OPE

**Date:** 2026-03-25
**Notebook:** `experiments/2026-03-25/MVP_v0.2.5.13_sigmoid_reward.ipynb`
**Builds on:** v0.2.5.6 (cross-policy guidance with hard threshold reward)

## Goal

Replace the hard binary reward (`cube_z > 0.84 -> 1, else 0`) with a smooth sigmoid reward to make OPE robust to diffusion noise in synthetic trajectories. The hypothesis was that small errors in generated `cube_z` near the threshold randomly flip binary rewards, and a sigmoid would produce proportionally small reward errors instead.

**Success criterion:** Spearman rho > 0.8 between guided sigmoid OPE and oracle SR.

## Setup

- **Diffuser:** Pre-trained on test checkpoint (54% SR behavior), EMA model from `diffusion_ckpts/mvp_v0252_traj_mse`
- **Guidance:** `score_timestep=5`, `action_scale=0.001`, positive-only, normalized gradients
- **Sigmoid:** `sharpness=150`, centered at 0.84
- **Generation:** 50 synthetic trajectories, T_GEN=60, chunk_size=4, 256 diffusion steps
- **6 target policies** evaluated (oracle SR: 8%–90%)

- Cube_z max distribution: mean=0.8424, std=0.0087, range=[0.8301, 0.8551]

## Analysis

**Result: FAIL** — sigmoid reward did not improve rank correlation (rho=0.49 for both hard and sigmoid, well below the 0.8 target).

### Why sigmoid didn't help
1. **The rank correlation is identical (0.4857)** for both hard and sigmoid scoring. The sigmoid smooths the reward magnitudes but doesn't change the rank ordering of policies. Since Spearman rho only cares about ranks, the sigmoid provides no benefit here.
2. **The guided OPE spread is tiny** — sigmoid OPE ranges only 7.153–7.739 across policies with oracle SR spanning 8%–90%. The guidance simply isn't differentiating policies enough for any reward model to recover the ranking.
3. **The sharpness sweep confirms this** — rho is flat across all sharpness values. The problem is not reward sensitivity; it's that guidance doesn't meaningfully steer trajectories differently for different target policies.

### Root cause (unchanged from v0.2.5.6)
The fundamental issue remains: guidance with `action_scale=0.001` barely affects the generated trajectories. All guided runs produce nearly identical cube_z distributions (SR ranges 58%–62%), regardless of whether the target policy has 8% or 90% oracle SR. The diffuser's prior dominates, and the guidance signal is too weak or poorly coupled to change the trajectory distribution.

### Comparison to v0.2.5.6
- v0.2.5.6 (hard only): rho = -0.2571
- v0.2.5.13 (hard): rho = +0.4857
- v0.2.5.13 (sigmoid): rho = +0.4857
- The improvement from -0.26 to +0.49 is likely due to run-to-run variance (different seeds / randomness), not the sigmoid. Both are far from the 0.8 target.

## Conclusion

Sigmoid reward is not the bottleneck. The core problem is that diffusion guidance doesn't differentiate between target policies — the generated trajectories look the same regardless of which policy guides them. Next steps should focus on making guidance effective (stronger coupling, different guidance mechanisms, or fundamentally different approach) rather than refining the reward model.
