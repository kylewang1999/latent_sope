## Goal

Add a dense staged reward function (reach + grasp + lift) alongside the existing hard/sigmoid rewards, and evaluate whether it improves rank correlation with oracle success rates.

## Dense Reward Design

Three-component weighted sum: `r = 0.3 * reach + 0.3 * grasp + 0.4 * lift`

- **Reach:** `exp(-10 * dist)` where `dist = ||gripper_to_cube||`
- **Grasp:** `sigmoid((-q1 + 0.02) / 0.005) * sigmoid((-dist + 0.03) / 0.01)` — fires when gripper is closed (q1 < 0.02) AND near cube (dist < 0.03m). Uses `q1` alone (index 17), not `q1+q2` as the design doc originally specified (q2 has opposite sign, so the sum is always ~0).
- **Lift:** `clip((cube_z - 0.8) / 0.1, 0, 1)`

## Analysis

### Dense reward performs worst for rank correlation

Dense reward's best rho is **+0.600** (scale=0.3), compared to hard reward's **+0.886** (scale=0.4) and sigmoid's **+0.943** (scale=0.3). The dense reward consistently underperforms both alternatives across all action scales.

### Why dense reward fails at ranking

The dense reward values are compressed into a narrow range (~15–19 across all policies) because the reach and lift components provide substantial baseline reward even for bad policies:
- **Reach component** fires whenever the gripper is near the cube, which happens in most trajectories regardless of policy quality
- **Lift component** gives partial credit for cube_z > 0.8, which is near the resting height (~0.82), so nearly all trajectories score > 0 on lift
- The **grasp component** does differentiate somewhat, but at 0.3 weight it can't overcome the shared baseline from reach + lift

By contrast, hard reward has a clear binary signal (cube_z > 0.84), and sigmoid concentrates its gradient right at the decision boundary, making both more sensitive to the success/failure distinction that determines oracle SR.

### RMSE is lower for dense (misleading)

Dense RMSE (0.37–0.39) looks better than hard (0.56–0.59) or sigmoid (0.50–0.54), but this is an artifact: the dense reward's narrow range means all predictions cluster near the mean, yielding low RMSE without actually ranking policies correctly. This is the classic "predict the mean" failure mode.

### State logRMSE degrades with scale (consistent with prior)

Mean state logRMSE goes from -2.42 (scale=0.1) to -1.86 (scale=0.5), confirming that higher guidance scales distort trajectory dynamics, consistent with v0.2.5.14d findings.

## Conclusion

Dense staged reward does not improve OPE for the Lift task. The reach and lift components provide too much shared baseline, compressing the reward range and destroying rank discrimination. The sigmoid reward (rho=0.943 at scale=0.3) remains the best reward function for this task.

**Recommendation:** Stick with sigmoid reward for Lift. Dense reward might help for tasks where the success/failure boundary isn't as clean, but for Lift (where success = cube_z > threshold), a reward function focused on that boundary outperforms a spread-out multi-component reward.
