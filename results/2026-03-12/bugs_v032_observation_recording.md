# Bug Report: Observation Recording & Reward Threshold (v0.3.2)

**Date:** 2026-03-12
**Discovered during:** MVP v0.3.2 multi-policy evaluation
**Severity:** Critical — causes all OPE estimates to be wrong
**Status:** Open

## Summary

Three related bugs cause the OPE pipeline to misidentify successful rollouts as failures. The root cause is that the rollout recorder stores pre-step observations, so the final state where the cube exceeds the lift threshold (0.84m) is never captured. This propagates into both target rollout SR computation and synthetic trajectory scoring.

---

## Bug 1: Rollout recorder drops the success state

**Location:** `src/latent_sope/robomimic_interface/rollout.py`, `rollout()` function (lines 462–498)

**Description:** The rollout loop records `obs` (pre-step observation) at each step, not `next_obs` (post-step). When the episode terminates on success, the loop breaks *before* `obs = next_obs` executes, so the final observation where `cube_z > 0.84` is never stored.

```python
for step_i in range(horizon):
    act = policy(ob=obs)
    next_obs, reward, done, info = env.step(act)  # cube crosses 0.84 HERE
    success = env.is_success()["task"]             # True

    recorder.record_step(
        obs=obs,         # pre-step obs, cube_z ≈ 0.838
        action=act,
        reward=reward,   # 1.0 (correct)
        next_obs=next_obs,  # post-step obs, cube_z > 0.84 (but store_next_obs=False)
    )

    if done or success:
        break            # breaks BEFORE obs = next_obs

    obs = deepcopy(next_obs)  # never reached on success step
```

**Evidence:** The 200demos_epoch40 policy (oracle 90% SR) has rollouts where:
- `reward_sum = 1.0` (env says success)
- `cube_z_max = 0.838` (< 0.84, the success threshold)
- Episodes are short (36–51 steps vs horizon=60), confirming early termination on success

Meanwhile, expert demos in the HDF5 file reach `cube_z = 0.864–0.886` because they store full post-action observations.

**Impact:** All recorded rollout trajectories lack the success state. Any downstream code that checks `cube_z > 0.84` on recorded data will never find a successful trajectory.

## Bug 2: Target rollout SR always reported as 0%

**Location:** `experiments/2026-03-12/MVP_v0.3.2_multi-policy_positive_and_negative_test.ipynb`, Cell 11

**Description:** The notebook computes target rollout success rate by checking max cube_z against 0.84:

```python
target_successes = [ep["states"][:, CUBE_Z_INDEX].max() > LIFT_THRESHOLD for ep in target_data]
```

Due to Bug 1, the max recorded cube_z is always ~0.838, so this always returns False. This is a cosmetic issue for the print statement, but it also means the pipeline has no signal about the true quality of target rollouts.

**Evidence:** All 4 policies report `Target rollout SR: 0.0%` even though env rewards show:
| Policy | Env SR (reward>0) | Threshold SR (z>0.84) |
|--------|-------------------|-----------------------|
| 50demos_epoch10 | 5% | 0% |
| 200demos_epoch10 | 20% | 0% |
| 10demos_epoch20 | 35% | 0% |
| 200demos_epoch40 | 90% | 0% |

## Bug 3: Synthetic trajectory scoring uses same broken threshold

**Location:** `experiments/2026-03-12/MVP_v0.3.2_multi-policy_positive_and_negative_test.ipynb`, Cell 9 (inline `score_trajectories_gt`)

**Description:** Synthetic trajectories are scored with:

```python
def score_trajectories_gt(trajectories, cube_z_index, threshold):
    cube_z = trajectories[:, :, cube_z_index]
    successes = np.any(cube_z > threshold, axis=1)  # threshold = 0.84
    returns = successes.astype(np.float32)
    return returns, successes
```

This uses the same 0.84 threshold. For synthetic trajectories generated by the diffuser, this is conceptually correct (the diffuser generates complete state sequences, not pre-step observations). However, the diffuser was **trained** on data that never reaches 0.84 (because of Bug 1), so generated trajectories are unlikely to exceed 0.84 either. The diffuser learns that "success" looks like cube_z ≈ 0.835, not 0.84+.

**Impact:** Even if the diffuser generates a trajectory that mimics a successful lift pattern, it will produce cube_z ≈ 0.835 (matching training data) and be scored as a failure. This systematically underestimates OPE for all policies.

**Note:** The library version of `score_trajectories_gt` in `src/latent_sope/eval/reward_model.py` uses `LiftRewardFn` with the same 0.84 threshold (`table_height=0.8, height_threshold=0.04`), so this bug also affects non-notebook usage.

## Root Cause Analysis

The fundamental issue is a mismatch between:
1. **robosuite's success condition**: checks `body_xpos` at end of `env.step()`, threshold 0.84m
2. **Recorded observations**: store `obs` from *before* the step, so the successful state is never captured
3. **Downstream scoring**: uses the same 0.84 threshold on data that can never reach it

Expert demos in the training HDF5 don't have this problem because they were recorded with a different pipeline that captures post-action observations.

## Proposed Fixes

### Option A: Fix the recorder (preferred)
Store the final `next_obs` when the episode terminates on success. This ensures the success state is captured in recorded data.

```python
if done or success:
    # Record the final transition so the success state is captured
    if recorder is not None:
        recorder.record_step(obs=next_obs, action=act, reward=0.0, done=True, next_obs=None)
    break
```

**Pros:** Fixes the root cause; 0.84 threshold remains correct; no downstream changes needed.
**Cons:** Requires re-collecting all rollouts.

### Option B: Use recorded rewards instead of cube_z threshold
For target rollout SR, use the env's reward signal (already recorded correctly):

```python
target_successes = [ep["rewards"].sum() > 0 for ep in target_data]
```

**Pros:** No re-collection needed; uses the authoritative signal.
**Cons:** Only fixes Bug 2; doesn't fix Bug 3 (synthetic trajectory scoring) or the training data issue.

### Option C: Lower the threshold for recorded data
Use a threshold like 0.83 for checking recorded observations, acknowledging the pre-step offset.

**Pros:** Quick fix.
**Cons:** Fragile — the gap between pre-step and post-step cube_z varies; doesn't fix the training data issue (diffuser still learns truncated trajectories).

### Recommended approach
**Option A + re-collect rollouts.** This is the cleanest fix and ensures the diffuser trains on data that includes the actual success states, which will improve generated trajectory quality.
